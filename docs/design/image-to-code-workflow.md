## 1. 目的

本文書は、AI を活用したフロントエンド開発において、ビジュアルデザイン工程とコーディング工程を分離し、高品質な成果物を効率的に得るための標準ワークフロー「Image-to-Code」を定義する。

## 2. 対象読者

- フロントエンドエンジニア
- デザインエンジニア
- AI エージェントを活用した開発を行う全メンバー

## 3. 前提知識

- ChatGPT（Images 2.0）の基本的な操作
- コーディングエージェント（Codex、Cursor、Claude Code 等）の基本的な使用経験
- React / Next.js によるフロントエンド開発の基礎知識

## 4. 背景と課題

AI に直接「モダンなサイトを作って」と依頼する従来のアプローチでは、以下の問題が発生する：

- 凡庸なデザイン（中央寄せヒーロー、グラデーション blob、弱い余白）が出力されやすい
- センス・レイアウト・アセット・レスポンシブ対応を同時に判断させるため、モデルの能力が分散する
- 出力の予測可能性が低く、反復調整のコストが高い

根本原因は、視覚的品質が技術以前の問題であるにもかかわらず、コード生成の文脈で全てを解決しようとしている点にある。

## 5. Image-to-Code とは

Image-to-Code は、コードではなくビジュアルリファレンスから開発を開始する手法である。

### 5.1 基本原則

```
画像が先、コードは後。
```

### 5.2 標準ワークフロー（6 ステップ）

| ステップ | 工程                   | 概要                                             |
| -------- | ---------------------- | ------------------------------------------------ |
| 1        | 画像生成               | 対象サイトのセクション単位のビジュアルを生成する |
| 2        | 方向性選択             | 生成結果から最適なビジュアル方向性を選定する     |
| 3        | アセット抽出           | 画像から個別のビジュアル要素を抽出・再生成する   |
| 4        | コード変換             | 画像とアセットをコーディングエージェントに渡す   |
| 5        | セクション構築         | フロントエンドをセクション単位で実装する         |
| 6        | スクリーンショット精査 | リファレンス画像とコード出力を比較し磨き込む     |

この分離により、モデルは視覚出力に集中でき、コーディング時には明確なターゲットが存在する。

## 6. 推奨ツール

### 6.1 画像生成

**ChatGPT Images 2.0** を第一推奨とする。

理由：

- レイアウト、階層構造、インターフェース構造、タイポグラフィの理解度が高い
- リファレンス画像を活用したスタイル指定が可能
- 反復探索のインターフェースがシンプルで制御しやすい

### 6.2 画像生成スキル

以下の再利用可能な Markdown スキルファイルを標準で使用する：

| スキル                   | 用途                                           | リンク                                                                                                 |
| ------------------------ | ---------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| imagegen-frontend-web    | ランディングページ、Web サイト、ダッシュボード | [SKILL.md](https://github.com/Leonxlnx/taste-skill/blob/main/imagegen-skills/frontend-web/SKILL.md)    |
| imagegen-frontend-mobile | アプリ画面、モバイルファースト製品             | [SKILL.md](https://github.com/Leonxlnx/taste-skill/blob/main/imagegen-skills/frontend-mobile/SKILL.md) |

スキルファイルは ChatGPT、Codex、その他画像モデルが利用可能な環境にコピーして使用する。プロジェクトのテイストに合わせてカスタマイズ可能。

### 6.3 コーディングエージェント

- Codex
- Cursor
- Claude Code
- 画像読み取りとプロジェクト編集が可能な任意のエージェント型環境

### 6.4 背景削除

- [Adobe Express 背景削除](https://express.adobe.com/home/tools/remove-background)（無料）

### 6.5 画像変換

- [cxnvert](https://cxnvert.vercel.app/)（WebP 変換、オープンソース・無料）

## 7. 詳細手順

### 7.1 ステップ 1：セクション単位の画像生成

**ルール**：フルページのスクリーンショット 1 枚で済ませないこと。

理由：ディテール不足、レイアウト判断の弱体化、正確な再現の困難化を招く。

**手順**：1 セクションにつき 1 枚の画像を生成する。

#### プロンプト例（スキル使用時）

```
上記のフロントエンド web 画像スキルを使用してください。
[プロダクト名/業界]のための、セクションごとのウェブサイトコンセプトを生成してください。
各セクションにつき1枚、別々の画像を作成してください：
1. ヒーロー
2. About
3. [セクション名]
...
スタイルの方向性：[方向性を具体的に記述]
ありきたりな SaaS のビジュアル、偽物のダッシュボードカード、
グラデーション blob、過剰にデザインされた AI 的な美意識は避けてください。
```

#### プロンプトの品質基準

| 評価       | 例                                                                                                         | 備考                                     |
| ---------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| **不十分** | 「かっこいいウェブサイトを作って」                                                                         | 具体性が皆無                             |
| **最低限** | 「クリーンな SaaS ランディングページを6セクションで」                                                      | 製品・業界の指定なし                     |
| **適切**   | 「学生向け AI ノートアプリ。ミニマル・構造的・やや遊び心。スタートアップの定型から外す。1セクション1枚。」 | プロダクト・業界・スタイル・ムードを含む |

### 7.2 ステップ 2：意味のあるスタイル指定

「モダン」等の空洞化したワードを避け、レイアウトとアートディレクションに影響する具体的な語彙を使用する。

#### 推奨スタイルワード例

- editorial、minimal、brutalist、swiss、organic
- structured、asymmetric、architectural、soft
- warm、technical、luxury、playful

#### 構図の指定例

- z-pattern、grid-based、magazine-layout、full-bleed
- split-screen、asymmetric-two-column

#### 「クリエイティブ」指定の注意

画像生成では「非常にクリエイティブに」が有効に働く場合がある（コーディングプロンプトとは逆）。使用時は必ず制約と組み合わせる：

```
クリエイティブに、ただしレイアウトは現実的で、プレミアム、
クリーン、かつ実際のフロントエンドで実装可能なものに。
```

### 7.3 ステップ 3：複数方向性の生成と選別

1. 同一プロンプトを異なるチャットで複数回実行
2. 出力を比較し、セクションごとに最良のものを選定
3. 例：チャット A のヒーロー + チャット B の機能セクション + チャット C の CTA
4. 選定後、直接ブラッシュアップを指示

### 7.4 ステップ 4：アセット抽出

生成されたウェブサイト画像に含まれる以下の要素を個別アセットとして抽出する：

- プロダクト写真、抽象 3D 形状、イラスト
- 背景テクスチャ、デバイスモックアップ、アイコン、装飾オブジェクト

#### 手順

1. 新規チャットを開き、セクション画像をアップロード
2. **「生成して抽出して」** と明示的に指示（「抽出して」のみだと単純な切り抜きになる）
3. アセットごとに繰り返す

```
このウェブサイトのヒーロー画像の右上にある抽象的な3Dオブジェクトを
生成して抽出してください。単体、クリーン、高解像度で。
```

#### 後処理

1. 透過が必要な場合は Adobe Express で背景を削除
2. WebP に変換（cxnvert を使用）

### 7.5 ステップ 5：画像からコードへの変換

**重要**：ウェブサイト全体を一括で変換しない。セクション単位で進める。

#### エージェントへの提供情報（セクションごと）

1. セクションのリファレンス画像
2. 当該セクションの抽出アセット
3. アセットファイル名一覧
4. レイアウト再現の明確な指示
5. 技術スタックとプロジェクト構造

#### 開始セクション

ヒーローセクションから着手する。

#### 初回出力の扱い

初回出力が完璧であることは稀である。これを前提とし、必ず次ステップの精査を行う。

### 7.6 ステップ 6：スクリーンショット精査

#### 手順

1. エージェントの出力をスクリーンショット撮影
2. リファレンス画像と並べて比較
3. 差異を具体的にフィードバック（矢印・丸・メモの描き込み推奨）
4. 修正指示をエージェントに返す
5. 全セクション完了まで 1-4 を繰り返す

#### フィードバック例

```
近いが、ヒーロー画像が小さすぎる。位置を上部に移動し、
サイズを約 20% 拡大。見出しベースラインをリファレンスに合わせる。
```

#### 精査ループ

```
リファレンス画像 → コード試行 → スクリーンショット → 視覚的フィードバック → 磨き込み → 次セクション
```

## 8. 品質チェックポイント

セクション実装時に以下の 6 項目を必ず確認すること：

| #   | 項目               | チェック内容                                                           |
| --- | ------------------ | ---------------------------------------------------------------------- |
| 1   | **配置**           | テキスト・カード・画像・グリッドが一貫したシステム上にあるか           |
| 2   | **余白**           | 窮屈さ、ランダムな隙間、要素の重なりがないか                           |
| 3   | **レスポンシブ**   | デスクトップのみでなくタブレット・モバイルを早期に確認したか           |
| 4   | **ブランド一貫性** | 色・タイポグラフィ・画像スタイル・ボタンが全セクションで接続しているか |
| 5   | **アセット品質**   | 画像・アイコン・3D オブジェクトの解像度と印象が十分か                  |
| 6   | **滑らかさ**       | レイアウトが意図的に構成され、セクション間の遷移が自然か               |

## 9. アニメーション追加

セクション構築完了後、以下を追加で実装する：

1. スクロールアニメーション（セクション進入時のフェードイン等）
2. ホバーアニメーション（ボタン・カード等のインタラクション）
3. レスポンシブの最終調整

プロンプト例：

```
新しいセクションにスクロール進入した際のアニメーションを追加。
スムーズかつクリーンな印象にすること。
```
