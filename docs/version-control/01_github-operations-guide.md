# GitHub運用ガイドライン

> 操作コマンドの詳細は `Git操作手順.md` を参照してください。

## 前提

- Next.js / TypeScript / App Router / Tailwind CSS / Supabase
- main シングルブランチ運用

---

## 1. 目的

このガイドラインは、Next.js / TypeScript / App Router / Tailwind CSS / Supabase を利用する小規模チーム開発において、日常開発からレビュー、マージ、リリースまでの Git / GitHub 運用を標準化することを目的とします。

このプロジェクトでは、**`main` を唯一の基準ブランチ**として扱います。

長期運用する `develop` ブランチは作成せず、すべての変更は `main` から作成した短命ブランチで行い、**Pull Request（PR）経由で `main` にマージ**します。

この運用の目的は、次のとおりです。

- 変更を小さく保つ
- レビューしやすくする
- コンフリクトを減らす
- `main` を常に信頼できる状態に保つ
- リリース判断を簡単にする

---

## 2. 基本方針

### 2-1. ブランチ運用方針

- 基準ブランチは `main` のみとする
- 作業は必ず `main` から作成した作業ブランチで行う
- 作業ブランチは短命とし、作業完了後は PR を作成する
- `main` への直接 push は禁止する
- `main` への反映は PR マージのみとする
- PR マージ後は作業ブランチを削除する

### 2-2. 運用対象

- アプリケーションコード
- Supabase 関連コード
- DB マイグレーション
- 型定義
- 設定ファイル
- ドキュメント

---

## 3. 必須ルール

- `.env.local` は絶対に push しない
- `SUPABASE_SERVICE_ROLE_KEY` をクライアントサイドで使用しない
- DB スキーマ変更は必ずマイグレーションとして管理する
- DB スキーマ変更時は型定義も更新する
- スキーマ変更・マイグレーション・型更新は原則同一 PR に含める
- `main` には直接 push しない
- PR を通さずに `main` を更新しない
- PR 作成前にローカル確認を実施する
- CI が失敗している PR はマージしない
- コミットメッセージは **プレフィックスは英語、説明は日本語** で記載する

---

## 4. ブランチ命名ルール

作業内容に応じて、以下のプレフィックスを使用します。

- `feature/`: 機能追加
- `fix/`: バグ修正
- `refactor/`: リファクタリング
- `chore/`: 雑務、設定変更、依存更新
- `docs/`: ドキュメント更新
- `test/`: テスト追加・修正

---

## 5. PR のルール

### 5-1. PR 作成の原則

- 1 PR 1 目的を原則とする
- PR はできるだけ小さく保つ
- 関係ない変更を混ぜない
- レビューしやすい単位で提出する
- 大きな変更は分割できるか検討する

### 5-2. PR タイトル

コミットメッセージと同様に、プレフィックスは英語、説明は日本語で記載します。

### 5-3. PR 本文に必ず書く内容

- 変更の目的
- 主な変更内容
- 確認方法
- 影響範囲
- 未対応事項がある場合はその内容
- DB 変更がある場合は migration / 型更新の有無
- UI 変更がある場合はスクリーンショット

---

## 6. レビューとマージのルール

- PR は少なくとも 1 名の承認を得てからマージする
- CI が成功してからマージする
- 指摘対応後は必要に応じて再レビューを依頼する
- 作成者本人のみの判断でマージしない
- 緊急対応を除き、レビューなしマージは行わない

### 推奨マージ方式

Squash merge を基本とします。

- `main` の履歴が見やすくなる
- 小規模チームでは細かい途中コミットをまとめやすい
- PR 単位で変更履歴を追いやすい

---

## 7. main の保護ルール

GitHub の Branch protection rules で、少なくとも以下を設定してください。

- `main` への直接 push を禁止
- Pull Request 必須
- 1 件以上の承認を必須
- CI 成功を必須
- force push を禁止
- branch deletion は必要に応じて制限

---

## 8. CI の最低基準

PR 時に、最低でも以下を CI で確認することを推奨します。

- `pnpm run lint`
- `pnpm run build`
- `pnpm exec opennextjs-cloudflare build`

可能であれば、将来的に以下も追加します。

- テスト
- 依存脆弱性チェック
- フォーマットチェック

---

## 9. Supabase / DB 変更時の運用ルール

DB 変更がある場合は、コード変更だけでなく、マイグレーションと型更新を含めて管理します。

### 必須ルール

- スキーマ変更は必ず migration file を作成する
- migration はアプリコードと同じ PR に含める
- 型定義も更新し、同じ PR に含める
- ローカルで migration 適用確認後に push する
- 変更内容が破壊的変更かどうかを PR に明記する

### DB 変更の対象例

- テーブル追加
- カラム追加
- 制約変更
- RLS ポリシー変更
- function / trigger 変更

### DB 変更時に確認すること

- ローカル環境で適用できるか
- アプリ側の型エラーが解消されているか
- 既存データへの影響がないか
- 本番適用時のリスクがないか

---

## 10. 認証・秘密情報のルール

- `.env.local` はコミットしない
- 秘密情報をコードに直書きしない
- `SUPABASE_SERVICE_ROLE_KEY` をクライアントサイドに渡さない
- サーバー専用の秘密情報はサーバー側のみで扱う
- 認証情報の漏えいが疑われる場合は、すぐにキーをローテーションする

---

## 11. リリース運用

小規模チームでは、`main` を常にリリース可能状態に保つことを基本とします。

### タグ付けルール

- タグは `main` のリリース対象コミットに付与する
- 基本は SemVer（例: `v1.0.0`）を推奨する
- 日付ベースで運用する場合は、チームで命名規則を統一する
- 誰がタグを切るかを決めておく
- hotfix 時の命名・手順も事前に決めておく

---

## 12. 認証運用の標準

このチームでは、GitHub 接続の標準を次のようにします。

- GitHub 接続は HTTPS を標準とする
- 認証は GitHub CLI を標準とする
- エディターは Visual Studio Code を標準とする
- パッケージマネージャーは pnpm を標準とする

SSH や別の認証方式を使うこと自体を完全禁止にはしませんが、チーム標準サポートは HTTPS + GitHub CLI とします。トラブル時は、まず標準手順で確認します。

---

## 13. この運用で重要な考え方

このガイドラインで最も重要なのは、次の 4 点です。

- `main` を常に壊さない
- 変更を小さく保つ
- レビューしやすい単位で出す
- 人の注意力ではなく、ルールと CI で品質を守る

---

## 14. まとめ

このプロジェクトでは、`main` を唯一の基準ブランチとするシンプルな運用を採用します。小規模チームでは、複雑なブランチ戦略よりも、短命ブランチ・PR・CI・ブランチ保護を徹底する方が、開発速度と品質のバランスを取りやすくなります。

全メンバーはこのガイドラインに従い、例外対応が必要な場合は PR またはチーム内で明示的に共有してください。
