# バージョン管理

バージョン管理は、変更履歴を安全に管理し、チームでの協業をスムーズにするための基盤技術です。

このプロジェクトでは **Git** を使用します。

---

## バージョン管理の基本

バージョン管理で学ぶべき主な項目は以下の通りです。

- Git の基本操作（リポジトリ作成、コミット、ステージング、差分確認）
- ブランチとマージ（並行開発の基礎）
- リモート操作（プッシュ、プル、フェッチ）
- GitHub の活用（PR、コードレビュー、Issue管理）
- コミットメッセージ（Conventional Commits、フォーマット）
- Git ワークフロー（GitHub Flow、Git Flow）
- 履歴の操作と回復（リセット、リバート、スタッシュ）
- pre-commit フック（自動チェック、品質保証）
- セキュリティとベストプラクティス（シークレット管理、.gitignore）
- トラブルシューティング（コンフリクト解決、エラー対応）

---

## ブランチ運用方針

このプロジェクトでは、**`main` を唯一の基準ブランチ**として扱います。

- 本番と同じ状態のブランチは `main` とする
- `dev` や `develop` などの長期運用ブランチは作成しない
- すべての変更は `main` から作業ブランチを切り、Pull Request（PR）経由で `main` にマージする
- 作業ブランチは短命にし、PR マージ後に削除する

### ブランチ命名規則

作業内容に応じて、以下のプレフィックスを使用します。

- `feature/`: 機能追加
- `fix/`: バグ修正
- `refactor/`: リファクタリング
- `chore/`: 雑務、設定変更、依存更新
- `docs/`: ドキュメント更新
- `test/`: テスト追加・修正

---

## 基本ルール

- `main` への直接 push は禁止する
- 作業は必ず `main` から作成した作業ブランチで行う
- 変更は Pull Request 経由で `main` にマージする
- PR は少なくとも 1 名の承認を得てからマージする
- CI が成功してからマージする
- コミットメッセージは **プレフィックスは英語、説明は日本語** で書く
- `.env.local` や秘密情報は絶対にコミットしない
- DB スキーマ変更は migration と型更新を同じ PR に含める

## 作業フロー

### 1. 作業開始

```bash
git switch main
git pull origin main
git switch -c feature/add-auth-pages
```

### 2. 実装とコミット

```bash
git status
git diff
git add app/login/page.tsx
git diff --cached
git commit -m "feat: Supabase認証ページを追加"
```

### 3. PR 作成前チェック

```bash
pnpm run lint
pnpm run build
pnpm exec opennextjs-cloudflare build
git push -u origin feature/add-auth-pages
```

### 4. Pull Request 作成

GitHub 上で `main` 向けの PR を作成し、レビューを依頼します。

### 5. マージと後処理

レビュー承認と CI 通過後、Squash merge で `main` にマージします。

```bash
git switch main
git pull origin main
git branch -d feature/add-auth-pages
```

## テスト環境

このプロジェクトでは、**Cloudflare Workers のローカルプレビューをテスト環境の標準**とします。

- 作業ブランチで `pnpm run preview` を実行し、Workers ランタイム上で確認する
- レビュアーは確認結果をもとにブラウザ確認を行う
- PR がマージされたら `pnpm run deploy` で Cloudflare Workers に反映する

本番環境は `main` へのマージ後に Cloudflare Workers へのデプロイで反映されます。

---

## 重要な考え方

バージョン管理で本当に大事なのは、すべてのコマンドを知っていることではありません。

- 小さく頻繁にコミットする
- 意味のあるコミットメッセージを書く
- ブランチを適切に使う
- コンフリクトを恐れない
- 履歴をきれいに保つ
- チームとルールを共有する
- セキュリティを意識する
- 問題が起きたら冷静に対応する
